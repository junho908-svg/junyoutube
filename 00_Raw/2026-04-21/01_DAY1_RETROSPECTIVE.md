## 1. 프로젝트 비전

### 처음 마음 (원문)

> "시간이 안나는 가족 구성원들이 재미있게 참여할 수 있고  
> 특히 어린이가 가정예배 통해서 성장하면서 신앙적으로 올바르게  
> 이 세상 살아갈 수 있고, 혹이나 다른 길로 가더라도  
> 가정예배를 통해서 다시 신앙생활 할 수 있는,  
> **돈으로 값어치 매길 수 없는** 웹과 앱을 만들어줘"

### 핵심 가치 4가지

| 가치 | 의미 |
|---|---|
| 🏠 **가족이 자연스럽게 모이게** | 부담 없는 5분 예배 |
| 🌱 **어린이가 즐겁게 배우게** | 게임 요소 + 예쁜 UI |
| ⛪ **신앙의 평생 자산** | 떠나도 돌아올 수 있는 등대 |
| 💝 **가치의 비영리성** | 돈으로 매길 수 없는 본질 |

### 타겟 사용자

- 자녀를 둔 30~50대 기독교 가정
- 가정예배를 시작하고 싶지만 막막한 부모
- 바쁜 일상 속 짧은 신앙 시간이 필요한 가족
- 말씀의지혜 채널 구독자 가족

## 4. 발생한 문제와 해결 과정

### 🔴 문제 1: BGM 호스팅 - Google Drive 실패

#### 시도와 실패 과정

**1차 시도**: Google Drive 직접 링크
```
URL: https://drive.google.com/uc?export=download&id=FILE_ID

❌ 실패 원인:
- CORS 미지원
- 리다이렉트 연쇄로 오디오 태그가 포기
- 범위 요청(Range) 제한
- 브라우저별 동작이 달라짐
```

**2차 시도**: Google Drive 새 형식
```
URL: https://drive.usercontent.google.com/download?id=FILE_ID&export=download

❌ 실패:
- Content-Type이 audio/mpeg가 아닌 application/octet-stream
- 일부 브라우저에서만 부분 작동
- 모바일 Safari에서 재생 실패
```

**3차 시도**: GitHub Raw URL
```
URL: https://raw.githubusercontent.com/USER/REPO/main/file.mp3

⚠️ 부분 성공:
- 파일은 다운로드되지만 스트리밍 불안정
- 한국 CDN 노드 없어서 느림
- GitHub이 용량 큰 파일 스트리밍 비권장
```

#### ✅ 최종 해결: GitHub + jsDelivr CDN

**4차 시도 (성공!)**:
```
URL 형식: https://cdn.jsdelivr.net/gh/USER/REPO@BRANCH/file.mp3

예: https://cdn.jsdelivr.net/gh/junho908-svg/family-worship-bgm@main/내모습이대로.mp3

✅ 장점:
✓ Content-Type: audio/mpeg (정확)
✓ Range 요청 완벽 지원 (탐색 가능)
✓ CORS 완벽
✓ 한국 CDN 노드 (서울)
✓ 무료, 무제한
✓ GitHub 자동 동기화 (파일 올리면 즉시 반영)
✓ Cloudflare + Fastly 더블 캐싱
```

#### 배운 교훈

> **"Google Drive는 파일 저장용이지 스트리밍용이 아니다."**

미디어 호스팅은 처음부터 다음 중 하나를 사용해야 한다:
- **GitHub + jsDelivr** (무료, 작은~중간 규모)
- **Cloudflare R2** (저렴, 대규모)
- **AWS S3 + CloudFront** (표준, 비용 있음)
- **전용 비디오 플랫폼** (Vimeo 등)

### 🔴 문제 2: 브라우저 자동재생 정책

#### 발생
앱이 열리자마자 BGM 자동 재생을 시도했지만 차단됨.
```
Error: play() failed because the user didn't interact with the document first.
```

#### ✅ 해결: 웰컴 게이트 디자인

**기술적 한계를 영적 의미로 전환**:
- 앱 첫 진입 시 "찬양과 함께 들어가기" 버튼 한 번 탭
- 이게 오히려 **"잠시 멈춰 주님 앞에 나아가는 거룩한 경계선"**이 됨
- 사용자 제스처 컨텍스트 안에서 즉시 play() 호출 → 정상 재생

**결과**: 불편함이 오히려 **앱의 핵심 UX**가 됨.

### 🔴 문제 3: GitHub README 충돌

#### 발생
```
Auto-merging README.md
CONFLICT (add/add): Merge conflict in README.md
Automatic merge failed; fix conflicts and then commit the result.
```

GitHub에서 자동 생성된 README와 로컬에 만든 README가 충돌.

#### ✅ 해결
```bash
git checkout --ours README.md    # 로컬 버전 선택
git add README.md                # 충돌 해결 표시
git commit -m "Resolve README conflict - keep app version"
git push
```

### 🔴 문제 5: 로고몽 이미지 캐싱

#### 발생
GitHub에 새 이미지 올렸는데 앱에서는 옛날 이미지 표시.

#### ✅ 해결 1: jsDelivr 캐시 강제 무효화
```
https://purge.jsdelivr.net/gh/USER/REPO@main/logomong.png
```

이 URL 한 번 방문하면 즉시 캐시 비워짐. 결과:
```json
{
  "status": "finished",
  "paths": {
    "/gh/junho908-svg/family-worship-bgm@main/logomong.png": {
      "throttled": false,
      "providers": {
        "CF": true,    // Cloudflare
        "FY": true     // Fastly
      }
    }
  }
}
```

#### ✅ 해결 2: URL에 버전 파라미터 추가
```javascript
// 이전
const LOGOMONG_URL = `.../logomong.png`;

// 이후
const LOGOMONG_URL = `.../logomong.png?v=3`;
```

파라미터 바꿀 때마다 브라우저가 새 리소스로 인식.

### 🔴 문제 6: React 변수 호이스팅 에러

#### 발생
```
Cannot access 'GITHUB_USER' before initialization
```

#### 원인
```javascript
// ❌ 잘못된 순서
const LOGOMONG_URL = `...${GITHUB_USER}...`;  // 먼저 사용
const GITHUB_USER = 'junho908-svg';           // 나중에 정의
```

JavaScript의 `const`는 호이스팅 안 되므로 **선언 전 사용 시 에러**.

#### ✅ 해결
```javascript
// ✅ 올바른 순서
const GITHUB_USER = 'junho908-svg';           // 먼저 정의
const LOGOMONG_URL = `...${GITHUB_USER}...`;  // 그 다음 사용
```
